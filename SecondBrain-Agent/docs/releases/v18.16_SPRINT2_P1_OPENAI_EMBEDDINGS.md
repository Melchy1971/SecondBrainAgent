# v18.16 Sprint 2 — P1 OpenAI Embedding SDK Integration

## Ziel

Der OpenAI Embedding Provider ist nicht mehr nur ein Stub. Er nutzt jetzt eine echte SDK-Grenze und kann production-ready werden, wenn SDK, API-Key und Health-Probe funktionieren.

## Geändert

```text
secondbrain/p1_embeddings.py
```

## Verhalten

### Erfolgsfall

Voraussetzungen:

- Extra installiert:

```powershell
pip install -e ".[openai]"
```

- API-Key gesetzt:

```powershell
$env:OPENAI_API_KEY="..."
$env:SECONDBRAIN_EMBEDDING_PROVIDER="openai"
```

Dann führt der Provider aus:

```python
OpenAI(api_key=...).embeddings.create(model=..., input=...)
```

Status:

```json
{
  "ok": true,
  "provider": "openai",
  "production_ready": true,
  "fallback_used": false,
  "semantic": true
}
```

### Fehlerfall

Wenn SDK, API-Key oder API-Call fehlen/fehlschlagen:

```json
{
  "ok": false,
  "production_ready": false,
  "fallback_used": true
}
```

`embed()` nutzt weiterhin lokalen deterministic Fallback, damit Entwicklung nicht hart abstürzt. Production Gates sehen aber den roten Provider-Status.

## Konfiguration

Über Environment:

```powershell
$env:SECONDBRAIN_EMBEDDING_PROVIDER="openai"
$env:SECONDBRAIN_EMBEDDING_MODEL="text-embedding-3-small"
```

Oder später über:

```text
config/vector_rag.yaml
```

Beispiel:

```yaml
embedding_provider: openai
embedding_model: text-embedding-3-small
```

## Tests

tests/test_v184_p1_embeddings_vectorstore.py

Neue Abdeckung:

- OpenAI Provider blockiert ohne API-Key
- OpenAI Provider blockiert ohne SDK
- OpenAI Provider nutzt SDK bei verfügbarem Fake-SDK

Die Tests nutzen ein Fake-`openai`-Modul und benötigen keinen echten API-Key.

Mit echter OpenAI-Integration:

```powershell

$env:OPENAI_API_KEY="..."
$env:SECONDBRAIN_EMBEDDING_PROVIDER="openai"
python launcher.py p1-embedding-status
```

## Erwartung

Ohne API-Key:

```text
p1-embedding-status -> provider.ok false, production_ready false
```

Mit gültigem API-Key + SDK:

```text
p1-embedding-status -> provider.ok true, production_ready true
```
