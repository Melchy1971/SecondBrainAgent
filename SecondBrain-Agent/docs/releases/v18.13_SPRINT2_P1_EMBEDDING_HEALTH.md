# v18.13 Sprint 2 — P1 Embedding Provider Health

## Ziel

P1 darf nicht mehr produktionsreif wirken, wenn die Embedding-Schicht nur mit lokalem Fake/Fallback läuft.

## Änderungen

### Provider Status

Geändert:

```text
secondbrain/p1_embeddings.py
```

Provider liefern jetzt aussagekräftige Statusfelder:

```json
{
  "ok": true,
  "production_ready": false,
  "semantic": false,
  "fallback_used": false,
  "mode": "local_test_double"
}
```

### Local Provider

`local-deterministic` bleibt für Entwicklung und Tests gültig:

```text
ok = true
production_ready = false
semantic = false
```

Damit kann `p1-gate` lokal weiterhin funktionieren, aber `p1-production` blockiert.

### Ollama Provider

`ollama` führt in `status()` einen echten Health-Probe gegen `/api/embeddings` aus.

Wenn der Endpoint nicht erreichbar ist oder kein valider Vektor kommt:

```text
ok = false
fallback_used = true
production_ready = false
```

Die Runtime kann für lokale Entwicklung weiterhin auf deterministic Embeddings fallbacken, aber Gates sehen den Fehler.

### OpenAI Provider

`openai` bleibt bewusst blockiert, bis die echte SDK-Integration umgesetzt ist:

```text
ok = false
production_ready = false
fallback_used = true
error = openai_embedding_client_not_implemented
```

## Production Gate

Geändert:

```text
secondbrain/p1_rag_runtime.py
```

`p1-production` prüft jetzt zusätzlich:

```text
embedding_provider_production_ready
```

Das Gate blockiert, wenn:

- `production_ready` false ist
- `fallback_used` true ist
- Provider-Status nicht ok ist

## Tests

Geändert:

```text
tests/test_v184_p1_embeddings_vectorstore.py
```

Neue Abdeckung:

- Local Provider ist deterministisch, aber nicht production-ready
- OpenAI Provider bleibt rot bis echter Client implementiert ist
- Embedding Status zeigt `production_ready = false` für lokalen Provider
- `p1-production` blockiert lokale Fallback-Embeddings
