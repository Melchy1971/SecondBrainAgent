### 2. Assistent

Ja, in Deutschland **müssen** öffentlich betriebene Webseiten – dazu zählen auch **Webseiten von Kindergärten in öffentlicher Trägerschaft (z. B. kommunal, städtisch, kirchlich)** – grundsätzlich **barrierefrei** sein.

### Gesetzliche Grundlagen:
1. **Behindertengleichstellungsgesetz (BGG)**  
   § 12a BGG verpflichtet öffentliche Stellen des Bundes zur Barrierefreiheit ihrer Internetangebote.

2. **Barrierefreie-Informationstechnik-Verordnung (BITV 2.0)**  
   Diese schreibt technische Anforderungen für barrierefreie Webseiten vor (z. B. Alternativtexte, Tastaturbedienung, Kontraste etc.).

3. **Landesrecht (z. B. L-BGG in Baden-Württemberg)**  
   Für Einrichtungen in kommunaler Trägerschaft (also auch kommunale Kitas) gelten in den Bundesländern zusätzlich **Landesgesetze**, die Barrierefreiheit im Web vorschreiben.

### Fazit:

| Trägerschaft       | Barrierefreiheit Pflicht? | Grundlage                         |
|--------------------|---------------------------|-----------------------------------|
| Kommunal / Stadt   | Ja                        | BGG, BITV 2.0, ggf. Landesrecht   |
| Kirchlich (öffentlich) | Ja                     | wenn über öffentliche Gelder finanziert |
| Privat             | Nein, aber empfohlen      | Freiwillig, aber benutzerfreundlich|

### Empfehlung:
Auch wenn ein Kindergarten privat ist, **empfiehlt** es sich dringend, zumindest grundlegende Barrierefreiheitskriterien einzuhalten – aus rechtlicher Vorsicht, Inklusionsgründen und weil viele Eltern mit Einschränkungen (z. B. Sehbehinderung, motorische Einschränkungen) die Seite nutzen könnten.

### **Python-Skript** (bereit zum Kopieren):

```python
import subprocess
import json
import pandas as pd
from datetime import datetime

# Ziel-URL
url = "https://www.kita-zaberfeld-michelbach.de"

# Pa11y-Aufruf
def run_pa11y(url):
    result = subprocess.run(
        ["pa11y", url, "--reporter", "json"],
        capture_output=True,
        text=True
    )
    return json.loads(result.stdout)

# Ergebnisse verarbeiten
def process_results(data):
    rows = []
    for issue in data:
        rows.append({
            "Typ": issue["type"],
            "Nachricht": issue["message"],
            "Kontext": issue["context"],
            "Selektor": issue["selector"],
            "Code": issue.get("code", "")
        })
    return pd.DataFrame(rows)

# Hauptlogik
data = run_pa11y(url)
df = process_results(data)

# Datei speichern
dateiname = f"pa11y_report_{datetime.now().strftime('%Y-%m-%d')}.xlsx"
df.to_excel(dateiname, index=False)
print(f"Bericht gespeichert unter: {dateiname}")
```

## 🔧 **Konkrete Verbesserungsvorschläge**

### 1. **Alternativtexte für Bilder (WCAG 1.1.1)**
- **Problem:** Einige Bilder haben keinen oder einen nichtssagenden `alt`-Text.
- **Lösung:** Alle dekorativen Bilder mit `alt=""`, alle inhaltlich relevanten Bilder mit beschreibendem `alt`-Attribut versehen.
- **Beispiel:**  
  ```html
  <img src="team.jpg" alt="Erzieherin Frau Müller beim Vorlesen in der Leseecke">
  ```

---

### 2. **Ausreichender Farbkontrast (WCAG 1.4.3)**
- **Problem:** Text auf hellen oder bunten Hintergründen ist möglicherweise schlecht lesbar.
- **Lösung:** Kontrastverhältnis für Text mindestens **4.5:1** (normaler Text), **3:1** (großer Text).
- **Tipp:** Mit [https://webaim.org/resources/contrastchecker](https://webaim.org/resources/contrastchecker) testen.

---

### 3. **Tastaturnavigation sicherstellen (WCAG 2.1.1)**
- **Problem:** Navigation und Fokussteuerung könnten ohne Maus schwer bedienbar sein.
- **Lösung:** Alle Menüs, Buttons und Links müssen per Tab erreichbar sein – mit sichtbarem Fokus (Outline).
- **Prüfung:** Mit der Tab-Taste durch die Seite gehen und sicherstellen, dass jedes Element angesprungen werden kann.

---

### 4. **Sprache der Website deklarieren (WCAG 3.1.1)**
- **Problem:** Wenn `lang="de"` im `<html>`-Tag fehlt, können Screenreader falsche Aussprache verwenden.
- **Lösung:**  
  ```html
  <html lang="de">
  ```

---

### 5. **Überschriftenhierarchie verbessern (WCAG 1.3.1)**
- **Problem:** Optisch hervorgehobener Text ist nicht immer semantisch korrekt als Überschrift ausgezeichnet.
- **Lösung:** Strukturierte Verwendung von `h1`, `h2`, `h3` usw. – keine Sprünge (z. B. von `h1` direkt zu `h4` vermeiden).

---

### 6. **Responsive Design (WCAG 1.4.10)**
- **Hinweis:** Auf Mobilgeräten scheint die Darstellung gut zu funktionieren, sollte aber regelmäßig getestet werden (z. B. auf Tablets, Zoom 200%).

---

### 7. **PDFs und Downloads barrierefrei gestalten (BITV Anforderung 11.3.1)**
- **Lösung:** PDFs mit Tags, korrekten Lesereihenfolgen und eingebetteter Sprache speichern – idealerweise durch barrierefreies Ausgangsdokument.

---

### 8. **Leichte Sprache und einfache Struktur (ergänzend empfohlen)**
- **Zielgruppe:** Viele Eltern (z. B. mit Deutsch als Zweitsprache) profitieren von klarer, einfacher Sprache.
- **Lösung:** Inhalte prüfen auf:
  - kurze Sätze
  - einfache Wörter
  - klare Gliederung
  - alternativ: eigene Seite in "Leichter Sprache"

---

### 9. **Erklärung zur Barrierefreiheit veröffentlichen (§ 12a BGG / § 3 BITV 2.0)**
- **Pflicht für öffentliche Stellen:** Eine Seite mit:
  - Stand der Barrierefreiheit
  - bekannten Problemen
  - Kontaktstelle für Rückmeldung (Feedback-Mechanismus)
