## Prozesslogik

1. Dateien rekursiv entdecken
2. Hash berechnen
3. Text extrahieren
4. Sprache/Titel/Summary/Keywords/Klassifikation erzeugen

## Kritische Schwächen und technische Risiken

### 1. Kein echter Migrationspfad als Systemstandard
Es gibt `migrations/init.sql`, aber das Backend setzt primär auf `Base.metadata.create_all()`.

Konsequenz:
- Schema drift wahrscheinlich
- Änderungen an bestehenden Tabellen schwer kontrollierbar
- Produktionsmigrationen werden unsauber
- `init.sql` und ORM können auseinanderlaufen

### 2. Embedding-Dimension ist inkonsistent modelliert
`chunks.embedding` ist im ORM flexibel als `Vector()` definiert.
`email_chunks.embedding` ist hart auf `Vector(1536)` gesetzt.
Im SQL sind Tabellen ebenfalls mit festen Dimensionen definiert.

Konsequenz:
Beim Wechsel zwischen Providern mit 768 und 1536 Dimensionen drohen harte Laufzeitfehler oder inkonsistente Datenhaltung. Das ist aktuell der größte strukturelle Entwurfsbruch.

### 3. Sprachmodellierung ist inkonsistent
- Dokumente nutzen englische Fulltext-Konfiguration
- E-Mails nutzen deutsche Fulltext-Konfiguration
- OCR/Dateien scheinen deutsch+englisch orientiert
- Sprache wird zwar erkannt, aber nicht konsequent zur Indexwahl verwendet

Konsequenz:
Trefferqualität hängt stark vom Dokumenttyp statt vom tatsächlichen Sprachinhalt ab.

### 4. Import-Session-Strategie ist gemischt
`routes_import.py` injiziert eine Session aus `get_db`, übergibt sie aber an `IngestionService(session_factory=lambda: db)`.
`IngestionService.import_folder()` erwartet eigentlich eine Session-Factory und öffnet pro Datei `async with self.session_factory() as session`.

Das funktioniert mit diesem Lambda nur deshalb, weil das Objekt bereits eine Session ist. Formal ist das ein Interface-Bruch.

Konsequenz:
- schwer testbar
- potenziell fragiles Session-Lifecycle-Verhalten
- Mehrdeutigkeit zwischen request-scoped und per-file scoped transaction handling

### 5. Runtime-Konfiguration speichert API-Keys lokal in JSON
`data/runtime_config.json`

Konsequenz:
- praktisch für lokal
- aber kein sauberes Secret-Management
- Risiko bei Desktop-/Shared-Systemen
- keine Trennung zwischen Betriebs- und Nutzerkonfiguration

### 6. Frontend enthält sehr große Seitenkomponenten
Beispiel: `ImportWizard.tsx` ca. 40 KB, `ChatPage.tsx` ca. 21 KB.

Konsequenz:
- hohe Änderungsreibung
- steigende Kopplung von UI-State, API-Logik und Rendering
- Regressionen bei Erweiterungen wahrscheinlich

### 7. Suchlogik und Chat hängen an derselben Retrieval-Qualität
Wenn Chunking, Embeddings oder Sprachindizes schwach sind, versagt nicht nur die Suche, sondern auch der Chat.

Konsequenz:
Chat-Qualität ist vollständig von der vorgelagerten Ingestion-Pipeline abhängig. Das ist richtig, aber operational relevant.

```text

  - `path`
```

```text
  - Zugriff außerhalb erlaubter Roots
  - symlink escape
```

```text
- Ersetze in `frontend/src/pages/ImportWizard.tsx` die bisherige Browse-Dialoglogik.

- Zeige:
  - Roots
  - aktuellen Pfad
  - Unterordner
  - Navigation eine Ebene hoch
  - Auswahl des aktuellen Ordners
- Übernimm den ausgewählten Pfad in `folderPath`.
- Lasse den bestehenden Scan-Flow danach unverändert weiterlaufen.

API-Client:
- Ergänze passende Methoden in `frontend/src/api/client.ts`.

Tests:
- Füge Tests hinzu für:
```

## Klare Einordnung

Die aktuelle Codebasis ist strukturell auf folgendes gebaut:

- **PostgreSQL**
- **SQLAlchemy ORM**
- **pgvector**
- relationale Modelle für:
  - `documents`
  - `chunks`
  - `email_messages`
  - `email_chunks`
  - `topics`
  - `chat_sessions`
  - `chat_messages`

**Neo4j ersetzt hier nicht nur die Persistenz**, sondern verändert das Datenmodell, die Query-Logik, die Suche, das Chat-Retrieval und große Teile des Importpfads.

## Konsequenz

Wenn du auf **Neo4j** umstellen willst, gibt es zwei sinnvolle Wege:

### Option A — vollständige Migration auf Neo4j
Vorteil:
- einheitliches Graphmodell
- Beziehungen, Entitäten, Themen, Referenzen, E-Mail-Netzwerke sehr stark

Nachteil:
- größter Umbau
- SQLAlchemy-Modelle werden weitgehend obsolet
- Suchlogik mit `pgvector` muss ersetzt oder neu gedacht werden
- Chunk-/Embedding-Speicherung und Retrieval müssen für Neo4j neu implementiert werden

### Option B — Hybrid-Architektur
- **Neo4j** für Wissensgraph / Beziehungen / Themen / Entitäten / Referenzen
- **Postgres + pgvector** bleibt für Dokumente, Chunks, Embeddings, Hybrid Search

Vorteil:
- deutlich geringeres Risiko
- bestehende Import-/Such-/Chat-Pipeline kann weiterlaufen
- Neo4j bringt schnell Mehrwert bei Beziehungen

Nachteil:
- zwei Datenbanken
- Synchronisationslogik nötig

## Fachlich beste Entscheidung für deine Codebasis

Für dieses Projekt ist **Option B kurzfristig die bessere Architektur**.

Begründung:
- Deine aktuelle Suche und RAG hängen direkt an Chunk-Embeddings
- Neo4j ist stark bei Graphbeziehungen, nicht automatisch Ersatz für eure aktuelle pgvector-basierte Retrieval-Architektur
- ein Vollwechsel würde sehr viel bestehende Funktionalität destabilisieren

Wenn du trotzdem **komplett auf Neo4j** willst, dann muss man das als **gezielten Architekturumbau** behandeln, nicht als Konfigurationsänderung.

## Was in Neo4j fachlich gut modellierbar ist

Geeignet als Knoten/Beziehungen:

### Knoten
- `Document`
- `Chunk`
- `Email`
- `Topic`
- `Person`
- `Organization`
- `Project`
- `Keyword`
- `ChatSession`
- `ChatMessage`

### Beziehungen
- `(:Document)-[:HAS_CHUNK]->(:Chunk)`
- `(:Email)-[:HAS_CHUNK]->(:Chunk)`
- `(:Document)-[:BELONGS_TO_TOPIC]->(:Topic)`
- `(:Document)-[:MENTIONS]->(:Person|:Organization|:Project)`
- `(:Email)-[:SENT_TO]->(:Person)`
- `(:Email)-[:SENT_BY]->(:Person)`
- `(:Chunk)-[:SEMANTICALLY_RELATED_TO]->(:Chunk)`
- `(:ChatMessage)-[:REFERENCES]->(:Chunk)`
- `(:Document)-[:DERIVED_FROM]->(:Email)`

Das ist stark für:
- Beziehungsanalyse
- Dokumentnetzwerke
- Themennavigation
- Quellenketten
- kontextuelles Retrieval über Graphnachbarschaft

## Was problematisch wird

### 1. ORM-Schicht fällt weitgehend weg
Die aktuelle SQLAlchemy-Modellierung passt nicht sauber auf Neo4j.

### 2. Suchlogik muss neu gebaut werden
Aktuell:
- Fulltext + Vektorsuche + Fusion

Mit Neo4j musst du entscheiden:
- nur Graph-Retrieval
- Fulltext in Neo4j
- Vektorsuche in Neo4j
- oder Hybrid mit externer Vektorhaltung

### 3. Importpfad muss neu persistieren
`ingestion_service.py` speichert aktuell relational.  
Das würde auf Graph-Writes umgebaut.

### 4. Chat-Service muss neues Retrieval-Modell nutzen
Nicht mehr SQL-Queries auf `chunks`, sondern:
- Cypher
- Graph-Retrieval
- ggf. Vektorindex + Graph Expansion

### 5. Migrationslogik wird komplett anders
Kein `init.sql`, kein `create_all()`, keine klassische relationale Migration.

## Technische Zielarchitektur bei Vollumstieg

### Backend-Schichten neu
- `Neo4jClient`
- `GraphRepository`
- `GraphIngestionService`
- `GraphSearchService`
- `GraphChatRetrievalService`

### Relationales Modell ersetzen durch Graphmodell
Beispiel:

statt:
- Tabelle `documents`
- Tabelle `chunks`

neu:
- `(:Document {id, title, source_path, hash, ...})`
- `(:Chunk {id, index, text, token_count, embedding, ...})`
- `(:Document)-[:HAS_CHUNK]->(:Chunk)`

### Query-Ebene
- SQLAlchemy Queries raus
- Cypher Queries rein
- evtl. Neo4j Python Driver oder offizieller Async Driver
